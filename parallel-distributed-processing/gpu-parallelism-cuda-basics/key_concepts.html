SIMT: threads, blocks, grids

A CPU has tens of complex cores optimized for sequential speed; a GPU has thousands of simple cores optimized for doing the same operation on different data. You write a kernel — a function that processes one element — and launch it over a grid of blocks of threads, one thread per element. Threads within a block can share fast memory and synchronize; blocks are independent (which is what lets the hardware scale). Your job is mapping thread ids to data indices: i = cuda.grid(1).

The transfer tax: host ↔ device memory

CPU (host) RAM and GPU (device) memory are separate; data crosses the PCIe bus, which is slow relative to GPU compute. This is the number-one reason naive GPU code disappoints: a fast kernel sandwiched between expensive copies. The strategies: keep data resident on the GPU across multiple operations, batch transfers instead of many small ones, and only ship back the (small) result. When you benchmark, time transfer and compute separately — the notebook does exactly this.

Amdahl's law: the serial ceiling

Speedup is capped by the fraction of the job that stays serial: with 20% serial work, infinite parallel hardware gives at most 5x. GPU-friendly problems are embarrassingly parallel — elementwise math, matrix multiply, image ops — where each output is independent. Branchy, sequential, pointer-chasing code doesn't map: threads in a group that take different branches execute both paths (divergence), serializing the parallelism you paid for.

5% serial → max 20x
20% serial → max 5x

The crossover point

For small arrays the CPU wins: kernel-launch latency and transfers dominate, and NumPy's C loops are already excellent. As size grows, the GPU's throughput advantage overtakes the fixed overhead — typically somewhere between 10⁵ and 10⁷ elements for simple elementwise ops. Finding this crossover empirically (the notebook's mini-project) builds the intuition that separates "GPU everything" enthusiasm from engineering. Corollary: batch small problems together so they land on the fat side of the crossover.

The Numba path (and the ecosystem)

numba.cuda compiles decorated Python functions to real CUDA kernels — the concepts (threads, blocks, device arrays) map one-to-one to CUDA C++, making it the ideal learning bridge. In production data work you'll more often use libraries that hide kernels entirely: CuPy (NumPy API on GPU), cuDF (pandas API), and PyTorch/JAX tensors. Learn the model with Numba so those libraries' performance behavior (why did this op force a sync? a copy?) stops being magic.
