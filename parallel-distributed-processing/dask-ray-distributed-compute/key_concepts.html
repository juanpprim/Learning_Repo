Dask DataFrame: chunked, lazy pandas

A Dask DataFrame is a collection of pandas DataFrames (partitions) plus a plan. Operations record tasks lazily; .compute() executes the graph across cores/machines and assembles the result. Because each partition is real pandas, the API is familiar — but the model leaks where an operation needs to see all data at once: sorts, groupby on non-index keys, and merge trigger expensive shuffles, exactly like Spark. Partition count is the tuning knob: aim for ~100MB partitions.

Task graphs & the dashboard

Every Dask computation is a DAG of small tasks. The scheduler assigns tasks to workers, respecting data locality (run the task where its input partition lives). The dashboard (started with Client(), usually port 8787) visualizes this live — task stream, memory per worker, transfer volume. Reading it is the debugging skill: lots of red (transfer) means your partitioning forces data movement; white gaps mean workers starved by scheduling overhead from too-tiny tasks.

Ray: tasks and actors

Ray parallelizes arbitrary Python, not DataFrames. Two primitives cover most needs:

Tasks — @ray.remote functions

  • f.remote(x) returns a future immediately
  • ray.get(futures) collects results
  • Stateless: perfect for embarrassingly parallel loops
  • Futures can chain: pass one task's output to another without gathering

Actors — @ray.remote classes

  • A live process holding state between calls
  • Method calls execute serially on the actor (no races on its state)
  • For model replicas, caches, counters, simulators
  • The reason ML serving frameworks build on Ray

Partitioning: the universal knob

The same tradeoff appears in Dask, Ray, and Spark. Too few chunks → idle workers and memory spikes; too many → scheduler overhead exceeds work. Corollaries worth internalizing: filter before shuffling operations; prefer building results per-partition and combining, over operations that need global order; and check df.npartitions after a big filter — 1000 near-empty partitions is a classic silent slowdown (fix with repartition).

When NOT to distribute

Distribution adds serialization, scheduling, network and debugging cost — it must be paid back. Before reaching for a cluster, exhaust the single-machine ladder: a better algorithm, vectorized NumPy/pandas, Polars or DuckDB (often 10x pandas on one machine), then more RAM — a 256GB cloud VM makes many "big data" problems small. Distribute when the working set truly exceeds one machine's memory, or one machine's runtime is hours you need to be minutes. "My laptop has 16GB" is a VM-sizing problem, not a cluster problem.
