The GIL: what it actually blocks

The Global Interpreter Lock lets only one thread execute Python bytecode at a time. Consequence: threads give zero speedup for CPU-bound pure-Python work — they take turns on one core. But the GIL is released during blocking I/O (network, disk, sleep) and inside most C extensions (NumPy inner loops), so threads work fine for I/O-bound tasks. This single fact drives every tool choice on this page. (Python 3.13+ has an experimental free-threaded build; the mainstream story is still the GIL.)

Threads interleave on CPU work but overlap on I/O waits

The decision rule

Classify the workload first, then the tool picks itself:

CPU-bound (hashing, simulation, parsing)

  • multiprocessing / ProcessPoolExecutor
  • Real parallelism: one GIL per process
  • Speedup ≈ number of physical cores
  • Threads help not at all here

I/O-bound (APIs, DBs, files)

  • asyncio for very many concurrent connections
  • ThreadPoolExecutor for a simpler drop-in fix
  • Cores irrelevant — you're overlapping waits
  • Processes work but waste memory

Process pools & the pickling tax

Processes don't share memory, so every argument and return value is pickled, sent through a pipe, and unpickled. This has two consequences: unpicklable objects (open connections, lambdas, locks) can't cross the boundary, and shipping large data can cost more than the computation saves. Chunk work so each task is coarse — milliseconds of compute per task drowns in overhead; seconds per task amortizes it well.

main: pickle args
pipe → worker process
compute (own GIL)
pickle result
pipe → main
Every crossing pays serialization — send small args, do big work

asyncio: one thread, thousands of waits

asyncio runs a single-threaded event loop. A coroutine runs until it hits await on something slow, then yields control; the loop resumes whichever other coroutine's I/O completed. No threads, no locks for most code, and concurrency limited by memory rather than thread count — 10,000 concurrent HTTP requests is routine. The discipline it demands: never block the loop. One synchronous time.sleep() or heavy computation freezes every coroutine; use await asyncio.sleep() and push CPU work to executors.

→

→

Shared state: race conditions

With threads, counter += 1 is read-modify-write — two threads can interleave and lose updates. Fixes in order of preference: don't share (each worker returns results; main aggregates — this is why executor.map is the safest pattern), use queues to pass ownership of data, or lock the critical section as a last resort. With asyncio, races still exist across await points but switches are predictable; with processes, sharing requires explicit machinery (Manager, shared memory) — the pickling wall protects you by default.
