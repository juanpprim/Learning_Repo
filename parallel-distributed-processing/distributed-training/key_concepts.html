The data-parallel idea

Every worker (process/GPU) holds a full replica of the model; each step, each worker gets a different shard of the batch, computes forward and backward locally, then all workers average their gradients so every replica applies the identical update and stays in lockstep. Effect: batch size scales with worker count, and epoch time shrinks proportionally — as long as gradient sync doesn't dominate.

The training step with all-reduce

All-reduce is the collective operation where every worker ends up with the sum (→ average) of everyone's gradients — implemented efficiently as ring all-reduce, where each node exchanges chunks with neighbors so bandwidth use is constant per node regardless of cluster size. DDP overlaps communication with computation: gradients for early layers are all-reduced while later layers are still doing backward, hiding much of the network cost.

The vocabulary: world size, rank, local rank

  • world size — total number of processes in the job (e.g. 2 nodes × 4 GPUs = 8).
  • rank — a process's global id, 0…world_size−1. Rank 0 conventionally does logging/checkpointing (guard side effects with if rank == 0 or you get 8 copies of everything).
  • local rank — id within one machine; used to pick the GPU: cuda:{local_rank}.
  • process group / backend — the communication layer: nccl for GPUs, gloo for CPU experiments. torchrun sets the env vars that wire this up.

DistributedSampler: shard, don't duplicate

Without it, every rank iterates the whole dataset — 4 workers doing 4x redundant work while gradients stay synchronized garbage-free but pointless. DistributedSampler gives each rank a disjoint 1/world_size slice. The classic bug: forgetting sampler.set_epoch(epoch), which makes every epoch use the same shuffle order (quietly hurting convergence). Second classic: per-GPU batch size × world size = effective batch size — scale the learning rate accordingly.

Beyond data parallelism

Data parallel (DDP)

  • Requires: model fits on ONE device
  • Scales throughput with more GPUs
  • Simple: wrap model, fix sampler, done
  • Where 95% of practitioners should start

Model / pipeline / sharded (FSDP, ZeRO)

  • For models too big for one device
  • Layers split across GPUs (pipeline) or params/optimizer state sharded (FSDP)
  • Communication-heavy, much harder to reason about
  • The LLM-training regime
Gradient accumulation: poor man's scaling — bigger effective batch on the same hardware, no extra GPUs
