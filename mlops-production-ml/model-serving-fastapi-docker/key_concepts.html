The request lifecycle

Every prediction request passes through the same pipeline. The critical idea: validation happens before your code runs. FastAPI parses the JSON body into your Pydantic model; if a field is missing or the wrong type, the client gets an automatic 422 with details, and your handler is never called with garbage.

Load the model once, at startup

Deserializing a model can take seconds; a prediction takes milliseconds. Loading inside the request handler multiplies your latency by 100x and hammers disk. Load into a module-level variable (or lifespan handler) so every request reuses the in-memory model.

Docker image layering

Each Dockerfile instruction creates a cached layer. Order matters: copy requirements.txt and pip install before copying your source code, so editing code only rebuilds the cheap final layer while the slow dependency layer stays cached.

  • .dockerignore keeps data files, venvs and .git out of the build context.
  • Multi-stage builds drop compilers/build tools from the final image.

Health checks: liveness vs. readiness

Orchestrators (Kubernetes, load balancers) need two different questions answered. Liveness: is the process alive at all? (if not — restart it). Readiness: is it able to serve correctly right now? (if not — stop routing traffic to it, e.g. while the model is still loading). A model server that answers "live" but hasn't loaded its model yet must answer "not ready".

Real-time vs. batch inference

Real-time (this subtopic)

  • Prediction on request, ms latency budget
  • Needed when input only exists at request time
  • Pay for always-on compute; must handle spikes

Batch

  • Precompute predictions on a schedule, store them
  • Serving becomes a cheap key-value lookup
  • Right answer surprisingly often — always ask this first
Rule of thumb: if you can enumerate the inputs ahead of time, batch it
