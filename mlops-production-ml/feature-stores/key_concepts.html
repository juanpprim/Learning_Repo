The problem: training/serving skew

The training pipeline computes avg_spend_30d in SQL; the serving service reimplements it in Python. The two drift apart — different null handling, different time windows — and the model quietly receives features distributed differently from what it trained on. A feature store fixes this by making the definition the single source of truth that both paths consume.

Without a feature store

  • Feature logic duplicated in 2+ codebases
  • Skew appears silently, found months later
  • Every team rebuilds the same features

With a feature store

  • Define once; training and serving both read it
  • Consistent transformations by construction
  • Features become a shared, discoverable catalog

Offline store vs. online store

The same feature values live in two places optimized for opposite access patterns. The offline store (parquet files, warehouse tables) holds full history for building training sets — huge scans, latency irrelevant. The online store (Redis, DynamoDB, SQLite locally) holds only the latest value per entity for serving — single-key reads in single-digit milliseconds.

Point-in-time correctness

The subtle killer when building training sets. For a training example labeled at time T, you must join the feature values as they were at T — using today's values leaks the future into training, inflating offline metrics that then collapse in production. Feast's get_historical_features does this as-of join for every row automatically; the notebook contrasts it with a naive (leaky) join.

Materialization

Materialization is the scheduled job that copies the freshest feature values from offline to online: feast materialize computes latest-per-entity and writes them to the online store. Its cadence sets your feature freshness — hourly materialization means serving reads features up to an hour stale, which is fine for slow features (30-day averages) and wrong for fast ones (items-in-cart), which need a streaming push path instead.

The vocabulary: entities & feature views

  • Entity — the business object features describe (driver, user), with a join key (driver_id).
  • Feature view — a named group of features from one source with a TTL, attached to entities.
  • Feature service — a versioned bundle of feature views that one model consumes; models depend on the service, not raw views.
  • TTL — how far back a point-in-time lookup may reach before the value is considered too stale to use.
