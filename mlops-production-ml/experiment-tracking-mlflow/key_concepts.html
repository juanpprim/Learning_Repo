Runs, experiments & what gets logged

A run is one execution of your training code; an experiment is a named bucket of runs you want to compare. Everything you log attaches to the active run, so a run becomes a complete, queryable record of "what did I try and what happened".

  • Params — inputs you chose: hyperparameters, dataset version, feature list. Immutable per run.
  • Metrics — outputs you measured: accuracy, loss, latency. Can be logged per step to form curves.
  • Artifacts — files: the serialized model, plots, sample predictions, the exact requirements.txt.
  • Tags — free-form metadata: git commit, author, "baseline"/"candidate".

Tracking backends: local files vs. database vs. remote server

Where the logs physically live determines what features you get. The code stays identical — only mlflow.set_tracking_uri(...) changes.

Local (./mlruns or sqlite:///mlflow.db)

  • Zero setup, perfect for solo learning
  • File store: no Model Registry
  • SQLite store: registry works, still single-machine

Remote server (DB + object storage)

  • Team-shared: everyone logs to one place
  • Postgres/MySQL for metadata, S3/GCS for artifacts
  • Enables access control and CI integration
Same API, different persistence — start local, move remote when a team needs it

Autologging vs. manual logging

mlflow.sklearn.autolog() patches the library so every fit() logs params, metrics and the model automatically. Manual logging gives you exact control over what is recorded. In practice you combine them: autolog for the bulk, manual log_metric for the business metric you actually care about.

Autolog

  • One line, captures ~everything the library knows
  • Logs internal params you didn't set explicitly
  • Great default; can be noisy

Manual

  • You choose the vocabulary of your experiments
  • Custom/business metrics (revenue lift, cost per error)
  • Needed for anything autolog can't see

Model Registry: versions, stages & aliases

The registry decouples "a good run happened" from "this model is what production uses". Registering a run's model creates a numbered version under a stable name (e.g. iris-classifier). Serving code asks for models:/iris-classifier@production and never hardcodes a run id — promoting a model is a metadata change, not a code deploy.

  • Versions are immutable; promotion moves a pointer, enabling instant rollback.
  • Modern MLflow prefers aliases (@staging, @production) over legacy stage transitions.

Why tracking is the foundation of MLOps

Every later MLOps practice assumes you can answer: which exact code, data and params produced the model currently serving? Tracking is what makes CI quality gates (compare candidate vs. baseline run), monitoring (compare live metrics vs. training metrics) and debugging ("what changed since v12?") possible. Without it you have vibes; with it you have lineage.
