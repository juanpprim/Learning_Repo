The three drifts

Models fail silently: the API returns 200s while predictions rot. Distinguish what moved:

  • Data drift — input distribution P(X) changed (new demographics, sensor recalibrated). Model may still be fine.
  • Concept drift — the relationship P(y|X) changed (fraud tactics evolved). Model is wrong even on familiar-looking inputs.
  • Label drift — the output distribution P(y) changed (base rate of positives moved).

Data drift is detectable immediately from inputs; concept drift usually needs (delayed) ground-truth labels — which is why input-drift monitoring is the early-warning system.

Measuring drift: PSI and KS

Both compare a reference window (training data) against a current window (recent production inputs), per feature.

  • PSI (population stability index): bin both samples, sum (cur% − ref%) · ln(cur%/ref%). Industry heuristics: <0.1 stable, 0.1–0.25 investigate, >0.25 act.
  • KS test: max distance between the two empirical CDFs; gives a p-value. Beware: with big samples, tiny meaningless shifts become "significant" — pair it with an effect-size threshold.

Rolling-window detection

Production monitoring is a loop: collect a window of recent traffic, compute drift stats vs. the reference, alert if thresholds are breached. Window size is the key tuning knob — small windows react fast but are noisy; large windows are stable but slow to fire. The notebook's mini-project makes this tradeoff visible.

Logs vs. metrics

Structured logs

  • One JSON record per prediction: inputs, output, latency, model version
  • High detail — lets you replay and debug single requests
  • Expensive to store/query at volume

Metrics

  • Pre-aggregated counters/histograms (Prometheus-style)
  • Cheap, fast to alert on, great for dashboards
  • No per-request detail
You want both: metrics to notice, logs to diagnose

Alerting without fatigue

An alert that fires weekly and is ignored is worse than no alert — it trains people to ignore all alerts. Alert only on conditions that are actionable (someone knows what to do), tier severities (page vs. ticket vs. dashboard), and periodically prune anything with a >50% false-positive rate. For drift specifically: alert on sustained drift across consecutive windows, not single-window blips.
