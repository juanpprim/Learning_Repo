Anatomy of a pipeline

A workflow is triggered by an event (push, PR, schedule), runs jobs on fresh runners, and each job is a sequence of steps. The ML twist: alongside code checks you add a step that exercises the model itself.

push
lint (ruff)
unit tests
train smoke-run
quality gate
build & deploy
A failure at any stage stops the flow — nothing broken reaches deploy

Code CI vs. model CI

Classic CI answers "does the code work?". ML pipelines must also answer "is the behavior still good?" — code can be perfectly correct while the model regresses because data or a dependency changed.

Code CI

  • Lint, type-check, unit tests
  • Deterministic pass/fail
  • Fast (seconds to minutes)

Model CI

  • Data schema validation
  • Training smoke test on a data sample
  • Metric thresholds vs. a pinned baseline
  • Slower; tolerate small nondeterminism with seeds/margins

The quality gate

The single highest-value ML test: train (or evaluate) the candidate and assert metric ≥ threshold. It converts silent model regressions into loud red CI. Two designs: a fixed floor ("accuracy must exceed 0.85") or a relative gate ("must not be worse than the current production model by >1%"). Fixed floors are simpler; relative gates catch slow decay.

Fast pipelines: caching

Slow CI silently stops being run. Cache the dependency install keyed on the hash of requirements.txt (actions/setup-python with cache: pip), keep CI datasets small and sampled, and split slow model jobs from fast lint/test jobs so quick feedback arrives first.

CI, CD… and CT

ML adds a third loop: Continuous Training. CI validates changes to code; CT re-runs training because the world changed (new data, drift alert), even with zero code changes. Mature setups trigger CT on a schedule or from a monitoring alert, and its output flows through the same quality gate and registry promotion as any human-made change.
