The game loop

Everything in a game happens inside one loop running dozens of times per second: read input, update the world one tick, render the new frame. This inverts normal program structure — nothing "waits" for the user; the loop always runs, and input is just state sampled each iteration. Almost every real-time system (games, robotics, trading engines, UI frameworks) is a variation of this pattern, which is why it's such transferable practice.

Delta time: frame-rate independence

If you move a player 5 pixels per frame, the game runs twice as fast on a 120Hz monitor as on 60Hz. The fix: measure the seconds since the last frame (dt) and express everything in per-second units: pos += speed_px_per_sec * dt. Now behavior is identical at any frame rate, and a lag spike causes one big-but-correct step instead of slow motion. (Physics engines go further with fixed timesteps + interpolation, but dt-scaling is the essential first tool.)

Per-frame movement (wrong)

  • pos += 5 each frame
  • 120fps machine → double speed
  • Lag → slow motion

Delta-time movement (right)

  • pos += 300 * dt
  • Same speed on every machine
  • Lag → bigger single step, correct total

Sprites & groups

A pygame.sprite.Sprite bundles the two things every game object needs: an image (what to draw) and a rect (where it is / its collision box). Groups hold sprites and give you bulk operations — group.update(dt) calls every sprite's update, group.draw(screen) blits them all. Architecturally this is composition over a giant loop of ifs: each object knows how to update itself, and the main loop just orchestrates groups (players, enemies, bullets).

main loop
players.update(dt)
enemies.update(dt)
bullets.update(dt)
all.draw(screen)
Groups let the loop stay tiny while object counts grow

Collision detection

The workhorse is axis-aligned rectangle overlap: rect.colliderect(other), or pygame.sprite.spritecollide(sprite, group, dokill) to test one against a whole group. Rect checks are fast but coarse — fine for most 2D gameplay. Refinements when needed: circle collision for round things, pygame.sprite.collide_mask for pixel-perfect (slow — use as a second pass after a rect hit). At scale, spatial partitioning (grids/quadtrees) avoids checking every pair — the same idea as database indexes.

broad phase: rect overlap?
narrow phase: mask/circle
resolve: damage, bounce, pickup
Cheap test first, expensive test only on candidates

Game states as a state machine

Menu, playing, paused, game-over are states: each has its own input handling, update logic and rendering, and transitions between them are explicit events (press Enter: menu → playing; hp ≤ 0: playing → game-over). Implement as a state variable dispatching to per-state functions, or a class per state. Without this, the loop degenerates into nested flag-checking spaghetti. This is the same finite-state-machine concept used for NPC AI in the game-ai subtopic — one pattern, many uses.
