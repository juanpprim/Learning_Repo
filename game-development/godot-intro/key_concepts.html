Everything is a node; scenes are trees

Godot's one big idea: a game is a tree of nodes, where each node does one thing (Sprite2D draws, CollisionShape2D defines a hitbox, Camera2D views). A scene is a saved subtree — a Player scene, a Bullet scene, a Level scene — and scenes instance other scenes, giving you composition all the way down. Children inherit their parent's transform: move the Player node and its sprite, collider and camera all follow.

Signals: decoupled events

Signals are Godot's observer pattern. Nodes emit ("body_entered", "timeout", "pressed") and other nodes connect handlers — the emitter never knows who's listening. This is what keeps scenes reusable: a Coin emits collected; whether the HUD, a sound player, or a quest system listens is not the Coin's problem. The rule of thumb the Godot community repeats: call down, signal up — parents call child methods directly; children communicate upward only via signals.

The lifecycle: _ready, _process, _physics_process

  • _ready() — runs once when the node enters the tree (children are ready before parents). Setup goes here.
  • _process(delta) — every rendered frame, variable rate. For visuals, UI, camera logic. delta is Godot handing you the dt from the pygame subtopic for free.
  • _physics_process(delta) — fixed rate (60Hz default). All movement and collision goes here so physics is deterministic and frame-rate independent.

Same loop as pygame — the engine just runs it for you and calls your hooks.

CharacterBody2D & move_and_slide

Godot ships the movement code you'd otherwise hand-roll. On a CharacterBody2D, set the velocity property (pixels/sec) and call move_and_slide(): the engine moves the body, detects collisions against physics layers, and slides along walls instead of sticking. Helpers like is_on_floor() make platformer logic trivial. Compare with the pygame subtopic where you built collision response manually — engines earn their complexity by packaging exactly this kind of hard-to-get-right code.

@export: code that designers can tune

Prefixing a variable with @export surfaces it in the editor's Inspector: @export var speed := 300.0 becomes an editable field on every instance of the scene. This separates mechanics (code) from tuning (data) — you can give three enemy instances different speeds without three scripts, and non-programmers can balance the game. It's the same design/config split you know from ML: hyperparameters in config, algorithm in code.
