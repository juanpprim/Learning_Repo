The log: topics & partitions

A Kafka topic is an append-only log split into partitions. Producers append; consumers read at their own pace. Two crucial properties: ordering is guaranteed only within a partition (the same key always routes to the same partition — so use e.g. user_id as key when per-user order matters), and partition count is the ceiling on consumer parallelism.

Consumer groups & offsets

A consumer group is a set of consumers sharing a group id; Kafka assigns each partition to exactly one consumer in the group, so adding consumers scales throughput — up to the partition count. Each group tracks its own offset (position) per partition, committed back to Kafka. This is why Kafka is pub/sub and a queue: different groups each get all messages; consumers within a group split them. When members join or leave, a rebalance reassigns partitions — brief pauses during rebalancing are normal.

topic (3 partitions)
group A / consumer 1 ← p0, p1
group A / consumer 2 ← p2
group B / consumer 1 ← p0, p1, p2 (all)
Groups are independent cursors over the same log

Delivery guarantees

Guarantees come from two knobs: producer acknowledgments and when consumers commit offsets.

  • At-most-once — commit before processing: crash loses the in-flight message. Fine for metrics ticks.
  • At-least-once — commit after processing: crash replays messages, so consumers must be idempotent (processing twice = processing once). The pragmatic default for data pipelines.
  • Exactly-once — idempotent producers + transactions; real but costly and mainly within Kafka-to-Kafka pipelines. In practice: at-least-once + idempotent sinks.

Producer side: acks=all waits for replicas — durable but slower; acks=0 is fire-and-forget.

Windowing: aggregating the infinite

A stream never ends, so "count per event type" is meaningless without a time boundary. Tumbling windows tile time into non-overlapping chunks (00:00–00:10, 00:10–00:20); sliding windows overlap (last 10 min, every 1 min); session windows close after a gap of inactivity. The deep complication: event time (when it happened) vs. processing time (when you saw it) — late-arriving events belong to windows you may have already emitted, which is what watermarks manage in Flink/Spark Streaming.

Streaming vs. batch

Batch (cron + files/warehouse)

  • Simple to build, test, backfill and reason about
  • Latency = schedule interval (minutes–hours)
  • Right when consumers only need periodic freshness

Streaming (Kafka)

  • Seconds of latency; events processed as they occur
  • Decouples producers from consumers (buffering, replay)
  • Operationally heavier: rebalances, ordering, duplicates
Start batch; move a pipeline to streaming when latency requirements force you to
