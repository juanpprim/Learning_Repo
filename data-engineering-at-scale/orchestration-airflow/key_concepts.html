DAGs, tasks & operators

A pipeline is a DAG — tasks plus dependency edges, no cycles. Each task is one unit of work created by an operator (PythonOperator, BashOperator, hundreds of provider operators for Spark/dbt/warehouses). Dependencies are explicit: extract >> transform >> load. The scheduler runs whatever is ready; independent branches run in parallel automatically — that's the point of declaring a graph instead of writing a script.

extract_api
transform
load_warehouse
notify
Airflow retries, monitors, and parallelizes the graph for you

Scheduling, catchup & backfill

Airflow's most misunderstood feature: a run for interval [Monday, Tuesday) executes after Tuesday begins — the run's logical_date is the interval start, not when it ran. With catchup=True, Airflow creates runs for every missed interval since start_date (deploy a daily DAG dated a year ago → 365 surprise runs; most DAGs want catchup=False). Backfill is using this machinery deliberately: re-running a date range after logic changes.

interval ends
→
run created (logical_date = interval start)
→
tasks execute
→
next interval
↻
Each run 'is about' its data interval, not its wall-clock time

Idempotency: the golden rule

Retries and backfills mean every task will eventually run twice. Tasks must produce the same result on re-run:

  • Overwrite, don't append — write to a path/partition derived from logical_date (/data/2026-07-02/) so re-runs replace instead of duplicate.
  • Delete-then-insert (or MERGE) for warehouse loads of the interval being processed.
  • No hidden state — process the run's interval, never "whatever is new since last time".

Non-idempotent DAGs make retries dangerous — which defeats the reason you adopted an orchestrator.

Passing data: XComs

XCom ("cross-communication") passes small values between tasks via the metadata database — a row count, a file path, a computed date. It is not a data plane: never push a DataFrame through it. The pattern for real data is pass references, not payloads: task A writes parquet to storage and XComs the path; task B reads the path.

task A → writes s3://…/out.parquet
XCom: just the path
task B → reads the path
Data moves through storage; XCom moves pointers

TaskFlow API vs. classic operators

Classic

  • Explicit Operator objects + PythonOperator callables
  • XCom push/pull is manual and stringly-typed
  • What most existing codebases contain

TaskFlow (@dag / @task)

  • Plain Python functions; decorators build the DAG
  • Return values become XComs; passing them draws the edges
  • The modern default for Python-centric pipelines
Same engine underneath — TaskFlow is ergonomics, not new semantics
