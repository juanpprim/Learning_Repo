Cluster anatomy: driver, executors, partitions

Your Python code runs on the driver, which plans work and hands it out. Executors are JVM processes on cluster nodes that do the actual computation. Data is split into partitions — the unit of parallelism: one task processes one partition, so 200 partitions can use at most 200 cores at once. Too few partitions = idle cores; too many tiny ones = scheduling overhead dominates.

The driver never touches the data; executors never see the whole dataset

Lazy evaluation and the DAG

Transformations (filter, select, groupBy) build a plan and do nothing. Only an action (count, collect, write) triggers execution. Laziness lets the Catalyst optimizer see the whole plan and rewrite it — pushing filters down before joins, pruning unread columns — before any byte moves. This is why df.explain() is your main debugging tool: it shows what will actually run, not what you wrote.

The shuffle: Spark's most expensive move

Operations like groupBy and joins need all rows with the same key on the same executor. Getting them there is the shuffle: every executor writes its data bucketed by key to disk, then every executor fetches its buckets from all others over the network. It's an all-to-all barrier — disk + network + serialization — and the boundary between stages in every query plan. Most Spark performance work is shuffle reduction: filter early, aggregate before joining, broadcast small tables.

Join strategies: broadcast vs. shuffle

Broadcast (map-side) join

  • Small table copied whole to every executor
  • Big table never moves — no shuffle at all
  • Force with broadcast(df); auto under ~10MB threshold
  • Fails if the 'small' table doesn't fit in memory

Shuffle (sort-merge) join

  • Both sides shuffled by join key
  • Handles two large tables
  • Vulnerable to key skew: one hot key → one giant straggler task
  • Fix skew by salting keys or AQE skew handling
Rule: if one side fits in executor memory, broadcast it

Columnar formats: why Parquet

CSV stores rows; Parquet stores columns, compressed, with min/max statistics per block. For analytics ("average of 2 columns out of 50") this changes everything: Spark reads only the needed columns (column pruning) and skips whole blocks whose stats can't match your filter (predicate pushdown). Partitioning output by a common filter column (write.partitionBy("date")) lets queries skip entire directories.
