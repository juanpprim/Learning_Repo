Columnar + MPP architecture

Two ideas explain warehouse speed. Columnar storage: each column stored (and compressed) separately, so SELECT AVG(amount) reads one column, not whole rows. MPP (massively parallel processing): a query is split across many workers, each scanning a slice, with partial results combined. This is also why warehouses are terrible at OLTP work — single-row updates fight the entire design; that's what row-store databases (Postgres) are for.

query: SUM(revenue) BY day
worker 1 — scans slice 1
worker 2 — scans slice 2
worker 3 — scans slice 3
combine partials
Scale comes from adding workers, not from indexes

Storage/compute separation

Coupled (classic databases)

  • Data lives on the query machines
  • Scaling compute means resharding data
  • One workload can starve another

Separated (BigQuery, Snowflake)

  • Data in cheap object storage; compute spun up on demand
  • Multiple independent warehouses over the same data
  • Pay for storage always, compute only when querying
This separation is what makes 'pay per query' pricing possible

Partitioning & clustering: scan less

Warehouses have no indexes; cost and speed are about scanning fewer bytes. Partitioning splits a table into chunks by a column (usually date) so WHERE date = ... touches one chunk. Clustering sorts data within partitions by other columns so block-level min/max stats let the engine skip blocks. The catch: pruning only works if the filter is directly on the partition column — WHERE DATE(ts) = '2026-07-01' wrapped in a function can defeat it.

Window functions: the analyst's power tool

OVER (PARTITION BY ... ORDER BY ...) computes per-row values that see other rows without collapsing them like GROUP BY does. The patterns worth memorizing:

  • ROW_NUMBER() … = 1 — deduplicate / latest record per key.
  • LAG()/LEAD() — compare to previous row: day-over-day deltas, session gaps.
  • SUM(x) OVER (ORDER BY d) — running totals; add a frame clause for rolling windows (ROWS BETWEEN 6 PRECEDING AND CURRENT ROW = 7-day rolling).
  • NTILE(n) — quantile bucketing.

The cost model

BigQuery on-demand charges by bytes scanned; Snowflake charges by compute time of running warehouses. Both reward the same habits: never SELECT * (columnar means every column named is money), always filter on partition columns, and dry-run first — BigQuery reports exact bytes a query would scan without running it (free). A dashboard on autorefresh against an unpartitioned table is the classic four-figure bill.
