Models are just SELECTs

A dbt model is one .sql file containing one SELECT. dbt wraps it in the DDL (CREATE VIEW/TABLE ... AS) and runs it against your warehouse in dependency order. You never write DDL, inserts, or "drop if exists" boilerplate — you declare what each dataset is, and dbt figures out how to build it. Version control, code review and CI then apply to your SQL like any other code.

ref() and source(): the dependency graph

Instead of hardcoding table names, models reference each other with {{ ref('stg_orders') }} and raw tables with {{ source('shop', 'orders') }}. This is dbt's core trick: from these references it builds the full DAG, so dbt run executes models in the right order, dbt run --select stg_orders+ rebuilds a model and everything downstream, and lineage documentation falls out for free.

Materializations

  • view (default) — no storage, always fresh, recomputed per query. Fine for staging.
  • table — fully rebuilt each dbt run. For marts queried often.
  • incremental — only processes new rows (WHERE ts > MAX(ts) in the target). Essential for large event tables; adds complexity around late data.
  • ephemeral — inlined as a CTE into consumers; exists nowhere in the warehouse.

Progression in practice: start everything as views, promote hot models to tables, go incremental only when rebuild cost forces you.

Tests: assertions about data

dbt tests are assertions compiled to SQL that must return zero rows. Built-in generic tests cover the big four — unique, not_null, accepted_values, relationships (foreign-key integrity) — declared in YAML next to each model. Custom tests are just SQL queries selecting violating rows. Run in CI, they catch upstream breakage (duplicate keys, new enum values) before dashboards go wrong.

Docs & lineage for free

dbt docs generate compiles a static site: every model's description (from YAML), columns, tests, compiled SQL, and — because of ref() — an interactive lineage graph from raw sources to final marts. Analysts can trace "where does this number come from?" without reading code. This discoverability is a large part of why teams adopt dbt beyond just "SQL with Jinja".
