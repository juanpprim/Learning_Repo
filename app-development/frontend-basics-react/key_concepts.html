Components, props, state — the mental model

React's contract: UI = f(state). A component is a function returning JSX; props are its inputs (passed by the parent, read-only); state is what it owns and may change. You never mutate the DOM — you call setState, React re-runs your function, diffs the result, and patches the real DOM minimally. Data flows one way (parent → child via props), which is what keeps large UIs traceable: to find why something renders wrong, follow the props up.

Hooks: useState & useEffect

useState gives a component a memory slot that survives re-renders; calling its setter triggers a re-render. useEffect handles side effects — anything beyond computing JSX (fetching, subscriptions, timers) — running after render, re-running only when values in its dependency array change: [] = on mount only. Two classic bugs to internalize early: a missing dependency (effect uses stale values) and setting state unconditionally inside an effect that depends on that state (infinite loop).

Controlled inputs

In a controlled form, React state is the single source of truth: the input displays value={name} and every keystroke goes through onChange → setName — the field literally cannot disagree with your state. This buys instant validation, easy programmatic reset (setName("") after submit), and derived UI (disable the button while invalid). Remember e.preventDefault() in the submit handler, or the browser does a full-page form post and unmounts your app.

Data fetching: model all three states

Every remote read has three outcomes and the UI must handle all of them: loading, error, success. Skipping the first two is how apps show blank screens and swallow failures. Pattern: a status state variable driving early returns. Also check res.ok — fetch does not reject on HTTP 4xx/5xx, only on network failure — and know that in real apps this whole concern usually moves into TanStack Query, which adds caching and retries on the same model.

Lifting state up

When two components need the same data (the form adds an item; the list shows it), the state moves to their closest common ancestor, which passes data down as props and hands mutation down as callbacks (onAdd). This is the answer to most "how do components talk?" questions. When the ancestor gets far away and props tunnel through many layers ("prop drilling"), reach for Context or a store — but not before: lifted state + props covers more than beginners expect, and premature global state is the classic React mess.
