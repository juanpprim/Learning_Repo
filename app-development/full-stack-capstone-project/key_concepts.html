Three-tier architecture

The capstone's shape: a React frontend that only talks HTTP/JSON, a FastAPI backend owning all logic and persistence, and behind it the data/model layer — the piece you pull in from an earlier track. The discipline that makes it work: the frontend never knows about the database or the model; it knows endpoints. That boundary is what lets you swap the model, change storage, or rewrite the UI independently.

The API contract comes first

Before writing either side, write down the endpoints: paths, request/response JSON shapes, error codes. This contract lets frontend and backend be built (and tested) independently — the frontend against mocks, the backend against TestClient. FastAPI generates OpenAPI docs at /docs automatically from your Pydantic models, so the contract is executable and always current. Contract-first is also how real teams parallelize across people; here it parallelizes across your own work sessions.

Model-as-endpoint

The ML integration pattern: load the trained model once at backend startup (from the MLflow registry if you built that subtopic), expose POST /predict validated by Pydantic, and persist each prediction (input, output, model version, timestamp) — giving the UI a history view and giving you the logging that the monitoring subtopic needs. One endpoint ties together serving, registry, and observability from the MLOps track.

Running the stack: compose & config

Multi-service dev is a container problem: docker compose up starts backend, frontend and database with one command, networked by service name. Two habits that make compose smooth: config via environment variables (API URLs, DB DSNs — never hardcoded, per twelve-factor), and dev-mode CORS on the backend so the Vite dev server's origin may call the API. Get compose working before deploying anywhere — most "works on my machine" disappears here.
