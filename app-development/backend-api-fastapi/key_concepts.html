REST: resources + verbs

Model your API as nouns (resources) manipulated by HTTP's standard verbs, and the whole design becomes predictable:

  • GET /items list · POST /items create · GET /items/42 read · PUT/PATCH /items/42 replace/update · DELETE /items/42 delete.
  • Status codes carry meaning: 200/201 success, 404 missing, 422 invalid body, 401 unauthenticated, 403 forbidden.
  • GET must be safe (no side effects); PUT and DELETE idempotent (repeat = same result) — the same idempotency idea as Airflow tasks and Kafka consumers.

Pydantic: schemas at the boundary

Declare the shape of requests and responses as typed models and FastAPI enforces them: invalid input gets an automatic 422 before your handler runs, and response_model filters what leaves. The pattern that matters: separate API schemas from database models — an ItemCreate (no id), an ItemRead (id, no internal fields), and the DB Item. Sharing one class couples your public contract to your storage, which is how password hashes end up in JSON responses.

raw JSON
parse + type-check
422 if invalid
handler gets typed object
response filtered by schema
Validation is declarative — the handler never sees bad data

Dependency injection with Depends

Depends is FastAPI's answer to "every route needs a DB session / current user / config": declare the need as a parameter (session: Session = Depends(get_session)) and the framework constructs and injects it per request, handling cleanup via generator dependencies. Dependencies compose — get_current_user itself depends on the token extractor — and tests override them (app.dependency_overrides[get_session] = fake) to swap in an in-memory DB without touching route code.

request → route handler
Depends(get_session) → DB session (auto-closed)
Depends(get_current_user) → verified user or 401
Depends(get_settings) → cached config
Declared needs, injected per-request, overridable in tests

Auth: password hashing + JWT

Two independent problems. Storage: never store passwords — store slow, salted hashes (bcrypt/argon2); verifying re-hashes the attempt and compares. Sessions: a JWT is a signed (not encrypted!) JSON payload — {sub: user_id, exp: ...} — that the server can verify statelessly on every request via its signature. Anyone can read a JWT's contents, so no secrets inside; and since tokens can't be revoked before expiry, keep lifetimes short.

POST /login
verify bcrypt hash
sign JWT {sub, exp}
client sends Bearer token
verify signature → user
Stateless auth: the token itself proves who you are

Migrations: versioning your schema

create_all() only creates missing tables — it will never alter an existing one, so the moment you add a column to a deployed app you need migrations: ordered, version-controlled scripts (Alembic) that transform the schema step by step, with a recorded current version per database. alembic revision --autogenerate diffs models vs. database and drafts the script — but always review it: autogenerate misses renames (seeing drop+add instead, which destroys data).
